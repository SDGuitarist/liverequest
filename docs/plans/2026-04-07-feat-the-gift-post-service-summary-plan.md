# feat: The Gift — Post-Service Summary PDF

## Enhancement Summary

**Deepened on:** 2026-04-07
**Research sources:** Context7 (@react-pdf/renderer docs), Security Sentinel, Performance Oracle

### Key Improvements from Deepening
1. **Bundle fonts locally** — Performance review identified Google Fonts CDN fetch as the #1 cold-start risk (300-1200ms per render). Bundling .ttf files in `public/fonts/` eliminates this entirely.
2. **Sanitize venue name in PDF filename** — Security review found Content-Disposition header injection risk (P2). Must strip special chars before setting filename.
3. **Promise.all for data queries** — 4 independent queries run in parallel (~150ms) instead of sequential (~600ms).
4. **GET route with cookie auth confirmed safe** — sameSite: lax allows cookies on top-level navigations (new tab, link click). CSRF not a risk for read-only GET.

### New Considerations Discovered
- Logo image should also be bundled locally (same CDN fetch risk as fonts)
- Single-performer assumption should be documented as a scope fence in the route (no gigId ownership check until multi-performer)
- History page must use a single aggregate query, not per-gig count fetches (N+1 risk)

## Overview

Auto-generate a Pacific Flow branded PDF summarizing gig performance data. Two access paths: (1) after debrief submission (toast with download link), (2) from a gig history page. Single `GET` API route handles PDF generation on demand. Data from Streams 1 (guest engagement) and 2 (musician intelligence), with graceful degradation for Stream 1-only gigs.

(See brainstorm: `docs/brainstorms/2026-04-07-cycle3-the-gift-brainstorm.md` for full scope decisions and rationale.)

## Problem Statement / Motivation

The Gift is LiveRequest's core go-to-market mechanism (Product Bible). After gigs, Alex delivers a post-service summary to venue contacts as an "unexpected gift" — building the relationship through data-backed insight. Currently this would require manually compiling data from Supabase. This cycle auto-generates the PDF from existing data.

## Proposed Solution

### Phase 1: PDF Generation API Route

**New file:** `app/api/gift/[gigId]/route.ts`

A `GET` route that:
1. Validates auth (`isAuthenticated()`) and `gigId` (UUID)
2. Queries all data for the gig (requests, sessions, song logs, venue name)
3. Renders a React PDF document via `@react-pdf/renderer`'s `renderToBuffer`
4. Returns the PDF as a `Response` with `Content-Type: application/pdf` and `Content-Disposition: attachment`

**Why GET (not POST):** A GET allows the browser's native download behavior — open in new tab, right-click save, direct link sharing. This is the first non-JSON API route in the codebase, establishing a new pattern for file responses.

**Route pattern:** `GET /api/gift/[gigId]` — dynamic segment for the gig ID.

### Phase 2: PDF Document Component

**New file:** `components/gift-pdf.tsx`

A React component tree using `@react-pdf/renderer` primitives (`Document`, `Page`, `View`, `Text`, `Image`, `StyleSheet`). Structured as:

```
GiftDocument (props: GiftData)
├── Header (Pacific Flow branding, venue name, date)
├── PerformanceOverview (config, duration, genre — only if Stream 2 exists)
├── GuestEngagement (total requests, response rate, peak timing)
├── TopRequested (top 5 songs by request count)
├── AudienceVibes (fire/energy/softer distribution)
├── PerformanceQuality (song quality/volume/acknowledgment — only if Stream 2)
├── Observations (debrief text fields — only if Stream 2)
├── WalkupsAndTips (walkup count, tips boolean — only if Stream 2)
└── Footer (Generated by LiveRequest for Pacific Flow)
```

**Graceful degradation:** When no Stream 2 data exists, sections are conditionally omitted (not rendered). The component checks `data.sessions.length === 0` to determine which sections to show.

### Phase 3: Data Aggregation

**New file:** `lib/gift-data.ts`

A single `getGiftData(gigId: string)` function that queries Supabase and returns a typed `GiftData` object:

```typescript
interface GiftData {
  gig: { venue_name: string; gig_date: string };
  requests: {
    total: number;
    played: number;
    responseRate: number; // played / total
    peakHour: string | null; // "8:00 PM"
    topSongs: { title: string; artist: string | null; count: number }[];
    vibes: { fire: number; more_energy: number; softer: number };
  };
  sessions: {
    setNumber: number;
    configuration: string;
    genreStyle: string | null;
    duration: number | null; // minutes
    songLogs: {
      total: number;
      quality: { off: number; fine: number; locked_in: number };
      volumeCorrect: number; // count where volume = 'right'
      guestAck: number; // count where guest_acknowledgment = true
    };
    debrief: PostSetData | null;
  }[];
  hasStream2: boolean; // true if sessions.length > 0
}
```

**Queries (all via `createServiceClient()` — performer-only route, `Promise.all` for parallelism):**

```typescript
const supabase = createServiceClient();
const [gigResult, requestsResult, sessionsResult] = await Promise.all([
  supabase.from("gigs").select("*").eq("id", gigId).single(),
  supabase.from("song_requests").select("*, songs(title, artist)").eq("gig_id", gigId),
  supabase.from("performance_sessions").select("*").eq("gig_id", gigId).in("status", ["complete", "post_set"]),
]);
// song_logs query depends on session IDs from sessionsResult
const sessionIds = (sessionsResult.data ?? []).map(s => s.id);
const logsResult = sessionIds.length > 0
  ? await supabase.from("song_logs").select("*").in("session_id", sessionIds)
  : { data: [] };
```

**Note:** 3 queries run in parallel (~150ms). The 4th (song_logs) depends on session IDs, so it runs after. Total: ~300ms for all data.

**Aggregation is server-side** — no shipping raw rows to the client. The `getGiftData` function computes all derived values (counts, percentages, peak hour) before returning.

**Peak hour calculation:** Group requests by hour of `created_at`, find the hour with the most requests.

**Top 5 songs:** Group by `song_id`, count, sort descending, take 5. If ties at position 5, include all tied songs.

### Zero-Data Contract

`getGiftData()` returns `GiftData | null`. It returns `null` only when the gig row does not exist.

For edge cases, the returned `GiftData` has these exact values:

| Scenario | requests.total | requests.responseRate | requests.peakHour | requests.topSongs | sessions | hasStream2 |
|----------|---------------|----------------------|-------------------|-------------------|----------|------------|
| Zero requests, zero sessions | 0 | 0 | null | [] | [] | false |
| Zero requests, 1+ sessions | 0 | 0 | null | [] | [...session data...] | true |
| 1+ requests, zero sessions | >0 | computed | computed | [...top 5...] | [] | false |
| Mixed sessions (complete + post_set) | aggregated | computed | computed | [...top 5...] | all sessions included | true |

**API route behavior for zero-data gigs:**
- Gig exists but has zero requests AND zero sessions → return **a degraded PDF** (header + "No guest engagement data was recorded for this event" message + footer). Do NOT return 404 or 409 — the gig is real, it just had no trackable activity. The Download button on the history page already hides for gigs with zero requests, so this path is rare (only reachable via direct URL).
- Gig does not exist → return **404**.

**Explicit computation rules:**
- `responseRate = total > 0 ? played / total : 0` (never NaN, never undefined)
- `peakHour = null` when there are no requests
- `topSongs = []` when there are no requests
- `vibes = { fire: 0, more_energy: 0, softer: 0 }` when there are no requests
- `session.debrief = null` when session status is `post_set` (debrief not submitted)
- **Top-5 aggregation filters out `song_id IS NULL`** before grouping (the schema has `song_id uuid NOT NULL` on `song_requests`, so this is a safety guard, not a functional requirement)

### Phase 4: Gig History Page

**New file:** `app/performer/history/page.tsx`

A server-rendered page showing completed gigs, auth-gated. Queries:

```sql
SELECT g.id, g.venue_name, g.gig_date,
  COUNT(DISTINCT sr.id) as request_count,
  COUNT(DISTINCT ps.id) as session_count
FROM gigs g
LEFT JOIN song_requests sr ON sr.gig_id = g.id
LEFT JOIN performance_sessions ps ON ps.gig_id = g.id
  AND ps.status IN ('complete', 'post_set')
WHERE g.is_active = false
  AND (sr.id IS NOT NULL OR ps.id IS NOT NULL)
GROUP BY g.id
ORDER BY g.gig_date DESC
```

**"Completed gig" definition (consistent with brainstorm):** `is_active = false` AND has at least one song request OR one performance session with status IN ('complete', 'post_set'). This matches the brainstorm's "any session or any request" rule. Gigs that were never started (no requests, no sessions) are excluded.

**Download button threshold:** At least 1 request OR 1 session with status IN ('complete', 'post_set'). This matches what the API can produce — a gig with only Stream 2 data (sessions but no guest requests) still generates a meaningful Gift with performance quality, observations, and walkups. Hiding the button for session-only gigs would contradict the API's capability.

**List items show:** Date, venue name, request count, session count, "Download PDF" link. The link is always shown for gigs on the history page (since the page query already filters to gigs with at least one request or session).

**No pagination for V1** — newest first, all gigs. Add pagination when count exceeds ~100.

**Session status narrowing (intentional):** The brainstorm says "any session" for history eligibility, but the plan narrows to `status IN ('complete', 'post_set')`. Sessions with status `pre_set` or `live` (abandoned mid-setup) are excluded because they have no meaningful data for the Gift.

### Phase 5: Post-Debrief Download Link

After debrief submission, the dashboard transitions to `complete` phase, which renders `PreSetForm` for the next set. Add a **conditional banner** at the top of the `complete`-phase PreSetForm:

```
┌─────────────────────────────────────────┐
│ ✅ Set complete! Download your Gift PDF │
│        [Download PDF]                   │
└─────────────────────────────────────────┘
```

**No state machine change** — the `complete` phase already shows `PreSetForm`. The banner is a conditional element that checks if a session just completed (passed via `previousSession` prop which already exists — see `dashboard/page.tsx:99`).

**Implementation:** The banner uses an `<a href={`/api/gift/${gig.id}`}>` tag (not a button with onClick) to trigger native browser download with cookie delivery. `gig.id` is already available via the existing `gig` prop — no new prop needed.

## Technical Considerations

### @react-pdf/renderer on Vercel

- **Runtime:** Node.js (default, confirmed — no Edge config in codebase)
- **API:** `renderToBuffer(<GiftDocument data={giftData} />)` → returns `Buffer`
- **Fonts:** Bundle .ttf files locally in `public/fonts/` (Outfit + Sora). Register via `Font.register({ family: 'Outfit', src: path.join(process.cwd(), 'public/fonts/Outfit-Regular.ttf') })`. **Do NOT use CDN URLs** — they add 300-1200ms on cold start. Font files are 50-200KB each.
- **Logo:** V1 uses text-only branding ("PACIFIC FLOW ENTERTAINMENT" as styled Text). No logo image asset exists yet. Adding a logo PNG is a future enhancement — when added, bundle in `public/images/` and verify the path on Vercel (same as fonts).
- **Emoji:** Use text labels (Fire, Energy, Softer) instead of emoji glyphs. `Font.registerEmojiSource()` fetches from Twemoji CDN — same cold-start risk. Emoji in PDFs is fragile across viewers.
- **Bundle size:** @react-pdf/renderer adds 2-5MB to function. Well within Vercel's 50MB limit.
- **Timeout budget:** Parallel data queries (~150ms) + renderToBuffer (~400ms) + overhead = ~600-800ms. With cold start: ~2-3s total. Comfortable under 10s free-tier limit.
- **Memory:** 10-30MB for a 1-2 page PDF. 1024MB default is ample.

### Response Pattern (new for codebase)

```typescript
// app/api/gift/[gigId]/route.ts
// NOTE: Single-performer assumption. Multi-performer requires JWT identity
// binding + gig ownership check (tracked as future work).

// Goal: header safety (no injection) + predictable downloadable filename.
// 1. Strip everything except word chars, spaces, hyphens, parens, dots
// 2. Collapse whitespace to single hyphens
// 3. Trim leading/trailing dots and dashes (avoids hidden files, confusing names)
// 4. Fallback to "venue" if sanitized result is empty
// 5. Cap at 80 chars (after fallback, before extension)
function sanitizeFilename(name: string): string {
  const cleaned = name
    .replace(/[^\w\s\-().]/g, '')
    .replace(/\s+/g, '-')
    .replace(/^[.\-]+|[.\-]+$/g, '')
    .trim()
    .slice(0, 80);
  return cleaned || 'venue';
}

export async function GET(request: NextRequest, { params }: { params: Promise<{ gigId: string }> }) {
  if (!(await isAuthenticated())) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
  const { gigId } = await params;
  if (!isUUID(gigId)) {
    return NextResponse.json({ error: "Invalid ID" }, { status: 400 });
  }

  const data = await getGiftData(gigId); // uses Promise.all internally
  if (!data) {
    return NextResponse.json({ error: "Gig not found" }, { status: 404 });
  }

  let buffer: Buffer;
  try {
    buffer = await renderToBuffer(<GiftDocument data={data} />);
  } catch (err) {
    console.error("PDF generation failed:", err);
    return NextResponse.json({ error: "PDF generation failed" }, { status: 500 });
  }
  const safeName = sanitizeFilename(data.gig.venue_name);
  const filename = `pacific-flow-${safeName}-${data.gig.gig_date}.pdf`;

  return new Response(buffer, {
    headers: {
      "Content-Type": "application/pdf",
      "Content-Disposition": `attachment; filename="${filename}"`,
      "X-Content-Type-Options": "nosniff",
    },
  });
}
```

### Security

- Auth-gated via `isAuthenticated()` (same as all performer routes)
- Uses `createServiceClient()` (performer-only, bypasses RLS)
- No new public endpoints — the Gift API and history page are performer-only
- **Cookie delivery on GET:** The `performer_auth` cookie (sameSite: lax) is sent on top-level navigations — `<a href>` clicks, `window.open()`, address bar entry. The download MUST be triggered by an `<a href="/api/gift/{gigId}">` tag or `window.open()` — these trigger native browser download behavior with cookies. Do NOT use `fetch()` — while same-origin `fetch` sends cookies, the response cannot trigger a native file download (would require blob URL + programmatic click, adding unnecessary complexity).

## Feed-Forward

- **Hardest decision:** Using GET instead of POST for the PDF route. Breaks the codebase's POST-only convention, but GET enables native browser downloads (open in new tab, right-click save). The trade-off is worth it for a file-serving endpoint.
- **Rejected alternatives:** Client-side PDF generation (avoids serverless constraints but exposes data to client bundle), caching generated PDFs (adds complexity for V1 — generate fresh each time), per-set breakdowns in PDF (valuable but complex layout — use gig-level aggregates for V1).
- **Least confident:** Whether `getGiftData()` handles all Supabase error states correctly under network instability. The fail-fast pattern (throw on query errors, catch in route) is sound, but untested under real failure conditions.
